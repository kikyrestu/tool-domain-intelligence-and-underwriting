# V1 + V1.1 Integrated Briefing
## Digital Asset Underwriting Engine

Dokumen tunggal untuk briefing programmer, delegasi kerja, monitoring progress, dan approval owner.

**Prinsip inti:** V1 tetap menjadi baseline MVP 30 hari; V1.1 diperlakukan sebagai hardening dan upgrade pada lapisan Source Discovery, provenance, dan pembuktian demo.

**Cara pakai singkat**
1. Kirim dokumen ini ke programmer sebagai source of truth implementasi.
2. Gunakan workbook master tracker sebagai dokumen hidup untuk update mingguan.
3. Owner approve berdasarkan demo, observed result, dan acceptance rule; bukan hanya status chat.

---

## 1. Tujuan Integrasi

Menggabungkan baseline V1 dan upgrade V1.1 menjadi satu paket implementasi yang lebih mudah didelegasikan ke programmer dan lebih mudah dipantau oleh owner. Dokumen ini menjaga scope MVP 30 hari tetap fokus pada discovery, availability, historical continuity, scoring, dashboard, dan export; sementara V1.1 dimasukkan sebagai penguatan terarah pada source discovery, parsing dokumen, normalization, provenance, dan quality gate demo.

---

## 2. Prinsip Integrasi

- V1 tetap menjadi tulang punggung produk: discovery → availability → historical continuity → scoring → dashboard → export.
- V1.1 bukan produk baru. V1.1 adalah hardening layer yang memperbaiki blind spot discovery agar tidak hanya HTML-centric.
- Owner hanya mengecek output, bukti demo, acceptance rule, observed result, blocker, dan dependency; tidak perlu membaca log teknis mentah.
- Programmer tidak boleh menandai task sebagai Done bila acceptance rule dan demo evidence belum terpenuhi.

---

## 3. Scope Terintegrasi

| Workstream | Asal | Isi Integrasi |
|---|---|---|
| Source Discovery | V1 | Input source URL, crawl dasar, ekstrak outbound links, dedup root domain. |
| Ingestion Router | V1.1 | Pisahkan parser HTML / PDF / DOCX / PPTX / text / unknown sejak awal pipeline. |
| Legacy HTML Support | V1.1 | Tolerant parser, relative URL resolver, meta refresh ringan, dan fallback plain-text URL. |
| Normalization & Provenance | V1.1 | Canonical URL, root domain, source type, parser type, source origin, extraction note. |
| Availability Check | V1 | Status actionable / watchlist / tidak actionable. |
| Historical Continuity | V1 | Snapshot dasar, bahasa dominan, drift kasar, continuity score. |
| Scoring Engine | V1 | Keputusan Buy / Review / Discard yang explainable. |
| Dashboard & Export | V1 + V1.1 | Shortlist owner-friendly dengan provenance terlihat di UI dan export. |
| QA Demo Matrix | V1.1 | Wajib ada demo tetap untuk HTML legacy, PDF, dan DOCX/PPTX. |

---

## 4. Batas Scope yang Tetap Ditunda

- Backlink intelligence mendalam per referring domain.
- Legal / trademark engine yang serius.
- Monetization fit engine yang kompleks.
- Desktop app lokal, scheduler skala besar, queue worker kompleks, multi-user permission, dan audit log tingkat lanjut.
- OCR berat atau semantic scoring kompleks sebagai syarat V1.1.

---

## 5. Alur Arsitektur yang Harus Dipatuhi

| Step | Layer | Aturan | Kenapa penting |
|---|---|---|---|
| 1 | Source Intake | Source URL atau file source masuk ke intake layer. | Tidak ada parser spesifik berjalan sebelum routing. |
| 2 | Content-Type Routing | Deteksi tipe input dan arahkan ke parser yang sesuai. | HTML/PDF/DOCX/PPTX/text/unknown tercatat. |
| 3 | Parser Execution | Parser menghasilkan extracted URLs + extraction note. | Parser modular, tidak patch liar di core flow. |
| 4 | Normalization | URL jadi absolute/canonical, root domain dihitung, dedup dilakukan. | Candidate bisa dibandingkan konsisten. |
| 5 | Provenance Logging | Simpan source type, parser type, source origin, timestamp, dan note. | Owner bisa audit asal setiap temuan. |
| 6 | Candidate Evaluation | Availability → continuity → scoring. | Lapisan V1 tetap utuh. |
| 7 | Owner Output | Dashboard, export, dan demo matrix. | Owner melihat hasil tanpa membaca log teknis. |

---

## 6. Integrated Workplan untuk Programmer

| Phase | Fokus | Exit condition |
|---|---|---|
| Phase 1 | Fondasi + scope freeze | Repo, env, DB, input source, scope V1.1 disetujui, tracker aktif. |
| Phase 2 | Router + parser modular | Content-type router dan modular extraction pipeline selesai dulu. |
| Phase 3 | Discovery hardening | Baseline HTML, legacy HTML, plain-text fallback, normalization, provenance. |
| Phase 4 | Document ingestion | PDF embedded links, PDF plain text, DOCX, PPTX. |
| Phase 5 | Evaluation engine | Availability, historical continuity, scoring tetap berjalan di candidate queue yang sama. |
| Phase 6 | Owner output + QA | Dashboard provenance, export provenance, fixed demo matrix, dokumentasi singkat. |

---

## 7. Acceptance Rules yang Mengunci Selesai

| Area | Rule |
|---|---|
| Content-Type Router | Minimal dua tipe source berbeda (mis. HTML dan PDF) masuk parser berbeda dan berakhir pada candidate queue yang sama. |
| Legacy HTML | Minimal satu halaman .asp/.aspx/.cfm atau HTML lawas menghasilkan outbound links valid yang sebelumnya gagal. |
| PDF Parser | Minimal satu PDF menghasilkan hyperlink aktif dan/atau plain-text URL yang masuk ke pipeline. |
| DOCX/PPTX Parser | Minimal satu DOCX atau PPTX mengekstrak link dari isi file, bukan sekadar URL file. |
| Normalization | Setiap candidate punya canonical URL, root domain, dan source origin. |
| Provenance | Detail candidate menampilkan source type, parser type, extraction note, dan timestamp. |
| Dashboard/Export | Owner dapat melihat provenance tanpa membuka log teknis. |
| QA Demo Matrix | Tiga demo pass: legacy HTML, PDF, DOCX/PPTX. |

---

## 8. Aturan Monitoring dan Delegasi

| Elemen | Aturan |
|---|---|
| Owner | Menetapkan prioritas mingguan, memberi approval scope, memutuskan blocker, dan menerima/menolak demo. |
| Programmer | Memecah task, update tracker minimal dua kali per minggu, menulis observed result, dan menunjukkan bukti demo. |
| Status task | Gunakan hanya: Not Started, In Progress, Blocked, Waiting Owner, Done. |
| Kolom wajib tracker | Task, expected output, acceptance rule, demo evidence, dependency, blocker, next step, observed result, owner verdict. |
| Definisi Done | Status boleh Done hanya jika acceptance rule terpenuhi, demo evidence tersedia, dan owner tidak meminta revisi. |

---

## 9. Struktur Dokumen dan Source of Truth

| Dokumen | Peran |
|---|---|
| Briefing terintegrasi ini | Dokumen utama untuk programmer. Menggantikan kebutuhan membaca V1 dan V1.1 terpisah dalam eksekusi harian. |
| Master tracker workbook | Dokumen hidup untuk monitoring, blocker, dependency, approval, dan change log. |
| Canon / manual lain | Hanya direvisi jika ada perubahan doctrine atau workflow operator, bukan untuk setiap patch parser. |

---

## 10. Instruksi Hari Pertama untuk Programmer

- Pilih stack final backend, database, dashboard, dan library parser yang realistis untuk 30 hari.
- Konfirmasi urutan dependency: router dan modular pipeline harus selesai sebelum claim support PDF/DOCX/PPTX.
- Isi tracker dengan target mingguan, acceptance rule, dan demo evidence yang akan ditunjukkan.
- Tandai risiko terbesar dan bagian yang sengaja tidak dikerjakan agar MVP tidak melebar.

---

## 11. Kesimpulan Operasional

Secara operasional, V1 memberikan kerangka produk dan V1.1 memperbaiki lapisan discovery agar lebih tahan terhadap source nyata. Paket final yang disarankan adalah: satu briefing terintegrasi ini + satu master tracker tunggal. Dengan cara ini owner cukup mengawasi satu brief dan satu tracker, sedangkan programmer mendapatkan urutan dependency, acceptance gate, serta bukti demo yang lebih tegas.
